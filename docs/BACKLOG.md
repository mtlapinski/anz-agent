# Backlog

Deferred items — not scheduled, but should be picked up by whoever (human or model) is next looking for follow-on work in this area.

- **2026-07-20 — Judge candidate pre-filtering.** The local search cache's judge subagent (see [docs/superpowers/specs/2026-07-20-search-cache-design.md](superpowers/specs/2026-07-20-search-cache-design.md)) currently receives *every* distinct cached query on each lookup, since cache entries never expire. Fine at small scale, but the judge prompt (and its cost/latency) grows unboundedly with usage. Needs a local pre-filter (e.g. shared-word heuristic) to shortlist candidates before invoking the judge. Lives in `tools/cache.py` / `tools/cache_judge.py`.
- **2026-07-20 — Fuzzy matches aren't persisted as aliases.** When a query fuzzy-matches an existing cached entry via the judge (`tools/cache.py`'s `lookup()`), the new query's results are returned but the new query itself is never stored. Repeating that same reworded query later still misses the exact-match fast path and re-invokes the judge every time, rather than being remembered as an alias of the original entry. Storing the new query as its own row pointing at the matched entry's results would collapse repeat lookups to the fast path, at the cost of propagating a wrong match if the judge ever errs. Compounds with the candidate-pre-filtering gap above at scale. Flagged in the final whole-branch review of the search-cache implementation.
